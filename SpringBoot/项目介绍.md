# 高校学科竞赛管理系统 — 完整项目介绍

> 答辩 & 面试用。一句话：**一个多角色的竞赛全流程管理平台，前后端分离，RESTful API。**

---

## 一、项目概述

- **项目名称**：高校学科竞赛管理系统
- **系统定位**：学生报名参赛、交作品、查成绩、申诉、报销；教师带队审核、打分、颁奖；管理员管全局

## 三、数据库设计（16 张表的关系）

```
user（用户统一表，role 区分 ADMIN/TEACHER/LEADER/STUDENT）
  ├── college（学院）
  ├── major（专业）
  ├── student_leader（学生↔领队关联，中间表）
  ├── registration（学生↔竞赛报名，中间表）
  ├── team（团队）
  │   └── team_member（团队→成员，一对多）
  ├── work_submission（作品提交）
  ├── score（成绩打分）
  ├── score_appeal（成绩申诉）
  ├── award（获奖记录）
  └── expense（报销记录）

competition（竞赛）
  └── competition_type（竞赛类型）

announcement（公告）
config（系统配置）
```

### 设计亮点：一个 user 表搞定四角色

- 传统设计分 admin、teacher、student 多张表，这里用 `role` 字段 + 空字段处理
- 学生用 `student_id` + `grade` + `major_id`，教师用 `teacher_id` + `specialty_field`
- 优点：登录统一查一张表，用户 ID 全局唯一，关联查询简单
- 代价：部分字段对特定角色为 NULL，靠业务代码保证一致性

## 四、角色权限体系

```
ADMIN（管理员）
  └── 全部 CRUD（用户、竞赛、学院、专业、公告、配置）

TEACHER（教师）
  └── 可申请成为 LEADER → 审核通过后拥有领队权限

LEADER（领队教师）← 关键角色
  ├── 审核学生关联申请
  ├── 审核报名
  ├── 打分、评价作品
  ├── 处理申诉
  ├── 审批报销
  └── 查看名下学生数据

STUDENT（学生）
  └── 报名、交作品、看成绩、申诉、报销
```

## 五、核心业务流程

### 学生参赛全链路

```
注册/登录 → 查看竞赛列表 → 选择领队老师（申请关联）
  → 老师审核通过 → 报名竞赛 → 组队（创建/加入团队）
  → 提交作品 → 等待评分 → 查看成绩 → 不满意可申诉
  → 获奖 → 报销费用
```

### 权限数据隔离逻辑（以 Score Appeal 为例）

```
ADMIN        → 查全部
TEACHER/LEADER → 只查 student_leader 表中自己名下的学生
STUDENT      → 只查自己
其他角色      → 返回空
```

## 六、关键代码设计

### 1. JWT 无状态登录（JwtUtil.java）

- 登录验证密码 → 生成 token（含 userId + username + role）
- 前端每次请求带 `Authorization: Bearer xxx`
- token 过期时间 24 小时

### 2. 统一响应 Result.java

### java
{
"code": 200,       // 200=成功, 400/401/403/404/500=各类错误
"message": "...",
"data": {...}
}
```

### 3. 报名时的业务校验（RegistrationController）

三重检查：
1. 参数完整
2. 不能重复报名
3. 必须已与领队建立正式关联

### 4. 作品撤回约束（WorkSubmissionController）

三把锁：
1. 已打分的不能撤回
2. 提交超过 24 小时不能撤回
3. 竞赛已截止不能撤回

## 七、项目评价

### 优点

- 功能完整：覆盖竞赛管理全流程
- 统一用户模型：单表四角色，简化认证和关联查询
- RESTful 设计：API 路径规范
- 权限数据隔离：通过关联表做行级数据隔离
- 前后端分离，方便独立部署

### 改进空间

1. **密码加密**：MD5 → BCrypt（Spring Security 的 BCryptPasswordEncoder）
2. **异常处理**：Controller 各自 try-catch → @ControllerAdvice 统一处理
3. **权限框架**：手写 token 解析 → Spring Security 或拦截器
4. **文件存储**：本地磁盘 → 阿里云 OSS / MinIO
5. **日志链路**：缺少 requestId 串联请求
6. **分页参数**：可封装通用分页工具，减少重复代码

## 八、答辩 Q&A 预案

**Q: 为什么用 MyBatis-Plus 而不是 JPA？**
> MyBatis-Plus 查询灵活，可手写复杂 SQL，分页插件开箱即用。JPA 在复杂多表关联时生成的 SQL 不可控。

**Q: JWT 过期了怎么办？**
> 前端检测 401 响应，自动跳转登录页。生产环境可加 refresh token 机制，校内使用 24 小时过期重登可接受。

**Q: 项目难点是什么？**
> 权限数据隔离。通过 `student_leader` 关联表做中间层，查询时根据角色动态拼接 SQL 条件，而不是简单写死判断。

**Q: 团队项目还是个人项目？**
> 个人独立完成前后端所有代码。
